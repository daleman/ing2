\section*{Introducci\'on}

Varios desarrollos dentro del \'area de la ingenier\'ia del software nos permite organizar la creaci\'on de sistemas complejos de una manera r\'apida y eficiente en el contexto de un equipo de m\'ultiples personas.

A continuaci\'on presentamos la simulaci\'on de un proyecto a desarrollar por 4 personas usando varias de estas t\'ecnicas. Con el conocimiendo adquirido en este estudio, este equipo hipot\'etico podr\'ia facilmente crear un software siguiendo los explicitados.

\section{Cronograma}

El proyecto est\'a separado en 4 iteraciones de 6 d\'ias-hombre cada uno, dando un total de 6 d\'ias de trabajo de 8 horas cada uno para cada persona y para cada iteraci\'on. En total, el proyecto deber\'ia terminar el \textbf{13 de junio}.

\begin{enumerate}[label={\LARGE\bf Iteraci\'on \arabic*: }, itemindent = 7em]
	\item {\LARGE\bf Elaboraci\'on (2 semanas)}
	\begin{itemize}
		\item Analizar sistema de streaming de partidos
		\item Analizando tecnolog\'ias para el sistema de pagos y extracciones
		\item Analizando tecnologías y riesgos en la obtenci\'on de los datos de marketing
		\item Diseñando fórmulas de posicionamiento (ranking)
		\item Diseñando estructura del log minuto a minuto
	\end{itemize}

	\item {\LARGE\bf Construcci\'on (3 semanas)}
	\begin{itemize}
		\item Accediendo a datos de usuario y de marketing
		\item Ingresando método de pago
		\item Retirando fondos
		\item Creando interfaz de comunicación con redes sociales
		\item Incorporando estadísticas de redes sociales y mensajes
		\item Viendo partido en vivo
	\end{itemize}

	\item {\LARGE\bf Construcci\'on} (3 semanas)
	\begin{itemize}
		\item Armando Equipo
		\item Armando Desafío
		\item Viendo progreso de desafío
		\item Aceptando desafío
		\item Mandando mensaje privado
		\item Mandando mensaje general
	\end{itemize}

	\item {\LARGE\bf Construcci\'on} (3 semanas)
	\begin{itemize}
		\item Viendo cuenta regresiva
		\item Extendiendo datos de simulación
		\item Accediendo al log minuto a minuto
		\item Visualizando simulación de partido
		\item Visualizando ranking
		\item Accediendo estado de la cuenta
	\end{itemize}

	\item {\LARGE\bf Transici\'on (2 días)}
	\begin{itemize}
		\item Restringiendo acceso/features según legislación
	\end{itemize}
\end{enumerate}

La primera iteración fue elegida teniendo en cuenta que los tiempos son
razonables y que la funcionalidad de cada caso de uso es importante por las
siguientes razones:

\begin{itemize}
  \item Armando equipo y armando desafío son la parte central de la aplicación. Además de eso, el frontend de estos casos de uso es importante para ver con los stakeholders y puede requerir futuros cambios. Armando desafíos además incluye la parte de poner una apuesta en el desafío, cosa que puede interesar a los stakeholders dado que es una potencial fuente de ganancia.

  \item Ingresando método de pago fue puesta en la primera iteración por dos razones. Primero, para mostrar a los stakeholders que la funcionalidad mínima necesaria para incorporar dinero a la aplicación (y una potencial fuente de ingresos) está siendo incorporada y testeada lo antes posible. Segundo, debido a su alto riesgo, al requerir mucha seguridad para evitar fraude financiero, es importante que sea estudiada y testeada desde un principio, para evitar potenciales pitfalls.
  \item Visualizando simulación de partido es una funcionalidad radicalmente nueva en la aplicación, y visualmente impactante, por lo que es útil de tener en una primera demo a los stakeholders.
  \item Accediendo al log minuto a minuto pertenece al core de la funcionalidad y los logs pueden ser de ayuda crítica al debugear siguientes etapas del desarrollo, además de ser un caso de uso de relativamente poco tiempo de desarrollo.
\end{itemize}

\section{Casos de uso con riesgo}

Hay varios casos de uso que consideramos que son riesgosos, dado que tienen complicaciones que deben ser revisadas con especial cuidado.

En particular,

\begin{description}
	\item [Accediendo los datos de los usuarios] es riesgoso por un tema de privacidad. Según la especificación dada por los stakeholders se piden los datos personales de los usuarios sin anonimizar, y esto puede traer ramificaciones legales.
	\item [Ingresando metodo de pago] tiene que tener una implementación que sea totalmente segura, o hay riesgo de que se cometa fraude y que haya problemas legales serios. Hay que tener especial cuidado a ataques que estén dirigidos al robo de información.
	\item [Incorporando estadísticas de redes sociales y mensajes] puede ser riesgoso ya que un usuario puede personificar a uno de estos servicios para mejorar si standing en el juego.
	\item [Restringiendo acceso/features según legislación] es riesgoso porque un error o la mala interpretación de una ley puede llevar a problemas legales inmediatos.
	\item [Viendo partido en vivo] es riesgoso desde el punto de vista de la disponibilidad, el streaming es un proceso demandante de recursos que es relativamente propenso a fallas.
	\item [Retirando fondos] es riesgoso porque hay una alta posibilidad de que usuarios intenten hacer fraude mediante este servicio.
\end{description}

\fbox{\parbox{\textwidth}{
	{\LARGE\underline{Riesgo 1}}
	\begin{description}
		\item [Descripción] Acceder a los datos de los usuarios es riesgoso por un tema de privacidad. Según la especificación dada por los stakeholders se piden los datos personales de los usuarios sin anonimizar, y esto puede traer ramificaciones legales.
		\item [Probabilidad] Alta
		\item [Impacto] Medio
		\item [Exposición]
		\item [Mitigación] Lograr que las queries de marketing solo respondan datos acumulados, y que no respondan nada si el sample es de pocos usuarios.
		\item [Plan de Contingencia] Cerrar temporariamente esta funcionalidad si se descubre la posibilidad de encontrar datos de un solo usuario.
	\end{description}
}}

\vspace{2em}

\fbox{\parbox{\textwidth}{
	{\LARGE\underline{Riesgo 2}}
	\begin{description}
		\item [Descripción] Puede haber una vulnerabilidad en el sistema de pagos que permita que se roben datos sobre las tarjetas de crédito de los usuarios.
		\item [Probabilidad] Media
		\item [Impacto] Alto
		\item [Exposición] Alta
		\item [Mitigación] Trabajar con expertos en seguridad infomativa para que el sistema de pagos sea totalmente seguro.
		\item [Plan de Contingencia] Suspender a los usuarios de los que se sospeche que están usando datos robados y avisar al usuario cuyos datos pueden haber sido robados.
	\end{description}
}}

\vspace{2em}

\fbox{\parbox{\textwidth}{
	{\LARGE\underline{Riesgo 3}}
	\begin{description}
		\item [Descripción] Parte de la simulación depende de sitios externos como cuentas de Twitter. Un usuario puede manipular estos sitios para mejorar su standing en el juevo.
		\item [Probabilidad] Baja
		\item [Impacto] Medio
		\item [Exposición] Baja
		\item [Mitigación] Normalizar los datos obtenidos por otros sitios para buscar cosas como usuarios twitteando sobre este jugador.
		\item [Plan de Contingencia] Suspender al usuario al que se descubra intentando hacer esto.
	\end{description}
}}

\vspace{2em}

\fbox{\parbox{\textwidth}{
	{\LARGE\underline{Riesgo 4}}
	\begin{description}
		\item [Descripción] Hay un riesgo de denegar (o no denegar) acceso a usuarios de manera equivocada basado en la legislación del país del mismo. Esto puede ser por haber implementado mal la legislación o que la misma cambie, lo último siendo muy probable.
		\item [Probabilidad] Media
		\item [Impacto] Alto
		\item [Exposición] Alta
		\item [Mitigación] Contratar o tercerizar los aspectos de la legislación a un experto del tema, que nos mantenga al tanto de los cambios.
		\item [Plan de Contingencia] Suspender temporariamente a los usuarios con los que haya problema hasta que haya sido aclarado como debe funcionar la legislación.
	\end{description}
}}

\vspace{2em}

\fbox{\parbox{\textwidth}{
	{\LARGE\underline{Riesgo 5}}
	\begin{description}
		\item [Descripción] Los servicios de streaming de TV online suelen ser poco confiables, y el sistema puede estar suspendido con relativa regularidad.
		\item [Probabilidad] Media
		\item [Impacto] Bajo
		\item [Exposición] Baja
		\item [Mitigación] Usar un servicio de backup en caso de que el servidio usado se caiga.
		\item [Plan de Contingencia] Darle links a los usuarios a servicios externos de streaming de TV hasta poder recuperar el usado en nuestra aplicación.
	\end{description}
}}

\vspace{2em}

\fbox{\parbox{\textwidth}{
	{\LARGE\underline{Riesgo 6}}
	\begin{description}
		\item [Descripción] Gente puede explotar la funcionalidad de retiro de fondos para conseguir ganar más dinero del que deberían tener por el sistema debido a errores en el sistema de pagos.
		\item [Probabilidad] Baja
		\item [Impacto] Alto
		\item [Exposición] Media
		\item [Mitigación] Asegurarse que el sistema de retiro de fondos sea seguro mediante una cerficación de seguridad externa.
		\item [Plan de Contingencia] Suspender a los usuarios perpetradores y arreglar los bugs encontrados.
	\end{description}
}}

\section{Descripción de Casos de Uso de la primera iteración}

\begin{enumerate}
  \item Accediendo a datos de usuario y de marketing: esto comprende armar una base de datos de cada
usuario con atributos pertinentes para hacer un análisis de cada uno como cantidad de horas en juego, cantidad de desafíos jugados, región del usuario, etc. Además dado que los datos los precisa un equipo de marketing se realizará una interfaz gráfica para visualizar los datos.
  \item Incorporando estadísticas de redes sociales y mensajes: este caso de uso agrega la funcionalidad de parsear los mensajes de los usuarios, tanto privados como generales, para utilizarlos en la evaluación de performance de cada jugador en los partidos simulados.
  \item Retirando fondos: este caso de uso realiza la funcionalidad de depositar una parte o la totalidad del dinero ganado en los desafíos a la cuenta bancaria del usuario. 
\end{enumerate}

\section{Lista de tareas relacionadas con los casos de uso de la primera iteraci\'on}

\begin{enumerate}[label=CU\arabic*]
	\item \textbf{Accediendo a datos de usuario y de marketing}
	\begin{detalle}
		\item Agregar usuarios a la base de datos
		\item Realizar interfaz gráfica para visualizar los datos del usuario.
		\item Realizar triggers de la base de datos para obtener información pertinente de cada usuario
		\item Hacer casos de test
		\item Correr los tests
		\item Corregir errores en base a tests
	\end{detalle}

	\item \textbf{Incorporando estadísticas de redes sociales y mensajes}
	\begin{detalle}
		\item Implementar parser de mensajes
		\item Implementar un scraper de redes sociales
		\item Hacer casos de test
		\item Correr los tests
		\item Corregir errores en base a tests
	\end{detalle}

	\item \textbf{Retirando fondos}
	\begin{detalle}
		\item Implementar front end para que el usuario pueda ingresar el numero de tarjeta de crédito, el código de seguridad y la fecha de vencimiento de la tarjeta
		\item Implementar el backend de la comunicación con la compañía financiera de la tarjeta de crédito para verificar si es una tarjeta valida
		\item Hacer casos de test
		\item Correr los tests
		\item Corregir errores en base a tests
	\end{detalle}

\end{enumerate}
